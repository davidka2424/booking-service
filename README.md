# Booking Service

Асинхронный бэкенд-сервис для записи на встречи с фоновой обработкой задач через очередь.

Пользователь создаёт бронь через REST API → запрос ставится в очередь → воркер асинхронно подтверждает запись (или симулирует отказ внешнего сервиса) → статус брони обновляется автоматически.

---

## Стек технологий

| Слой | Технология | Почему |
|---|---|---|
| API-фреймворк | FastAPI 0.115 | Async из коробки, автодокументация, Pydantic-валидация |
| Очередь задач | TaskIQ + Redis broker | Полностью async, единый event loop с FastAPI (см. раздел "Технические решения") |
| База данных | PostgreSQL 16 + SQLAlchemy 2.0 (async) | Промышленный стандарт, нативная поддержка async через asyncpg |
| Миграции | Alembic | Версионирование схемы БД |
| Логирование | structlog | Структурированные JSON-логи в продакшне |
| Rate Limiting | slowapi | Защита POST-эндпоинта от перегрузки |
| Тесты | pytest + httpx + aiosqlite | Async-тестирование без поднятой инфраструктуры |
| Линтер / типизация | ruff + mypy | Статический анализ кода |
| CI | GitHub Actions | Автоматическая проверка lint + типов + тестов на каждый push |

## Быстрый старт — просто запустить проект

Нужен только Docker и Docker Compose, больше ничего устанавливать не требуется.

```bash
git clone https://github.com/davidka2424/booking-service.git
cd booking-service
cp .env.example .env
docker-compose up --build
```

Одна команда поднимает весь стек:
1. PostgreSQL и Redis
2. Alembic-миграции (применяются автоматически перед стартом API)
3. FastAPI-сервер на `http://localhost:8000`
4. TaskIQ-воркер, который слушает очередь и обрабатывает брони в фоне

### Проверка что всё работает

```bash
curl http://localhost:8000/health
# {"status":"ok","env":"development"}
```

Интерактивная документация (Swagger UI): **http://localhost:8000/docs**

Через Swagger можно создать бронь, посмотреть её статус, отменить — без единой строчки кода.

## Если нужно разрабатывать проект дальше

### Запуск с hot-reload (без пересборки образа на каждое изменение)

```bash
docker-compose -f docker-compose.dev.yml up --build
```

Код монтируется как volume — после первого запуска любое изменение в `src/` подхватывается автоматически (API через `--reload`, воркер через `--fs-reload`). Пересборка образа нужна только если меняешь `requirements.txt` или `Dockerfile`.

### Локальное окружение для разработки (опционально)

Если хочешь работать с кодом в IDE с автодополнением, типизацией и линтером без захода в контейнер:

```bash
python -m venv .venv
.venv\Scripts\activate        # Windows
# source .venv/bin/activate   # Linux/macOS

pip install -r requirements-dev.txt
```

Это не обязательно для запуска проекта — только для удобства разработки. Docker самодостаточен и не зависит от локального `.venv`.

### Применение миграций вручную

```bash
docker-compose run --rm migrate alembic upgrade head
```

### Создание новой миграции

```bash
alembic revision --autogenerate -m "описание изменения"
```

## API

### Создать бронь
```bash
curl -X POST http://localhost:8000/api/v1/bookings \
  -H "Content-Type: application/json" \
  -d '{
    "name": "Иван Петров",
    "scheduled_at": "2099-06-01T10:00:00+03:00",
    "service_type": "consultation"
  }'
```

Доступные значения `service_type`: `consultation`, `repair`, `installation`, `maintenance`, `inspection`

### Получить статус брони
```bash
curl http://localhost:8000/api/v1/bookings/{id}
```

Жизненный цикл статуса: `pending` → `confirmed` / `failed` (по итогу обработки воркером), либо `pending` → `cancelled` (по запросу пользователя).

### Список броней (с фильтрацией и пагинацией)
```bash
curl "http://localhost:8000/api/v1/bookings?status=pending&page=1&size=20"
```

### Отменить бронь
```bash
curl -X DELETE http://localhost:8000/api/v1/bookings/{id}
```

> Отменить можно только бронь в статусе `pending`. При другом статусе вернётся `409 Conflict` с понятным сообщением.

## Тесты

**Текущий статус: 28/28 тестов проходят, ~77% покрытия кода.**

Тесты покрывают:
- Все 4 эндпоинта API (happy path + граничные случаи: невалидные данные, несуществующие записи, повторные операции, пагинация)


Тесты используют SQLite in-memory вместо PostgreSQL и мокают TaskIQ-брокер — никакой внешней инфраструктуры не требуется. Это соответствует требованию задания: тесты должны запускаться `pytest` из корня без поднятого Docker.

## CI/CD

При каждом push в репозиторий автоматически запускается GitHub Actions ([`.github/workflows/ci.yml`](.github/workflows/ci.yml)), который последовательно проверяет:

1. **Lint & Type Check** — `ruff check`, `ruff format --check`, `mypy`
2. **Tests** — полный прогон pytest с покрытием кода

Если любой из шагов падает — pull request получает красный крест, мерж не происходит. Статус текущего билда отображается значком в начале этого README.

## Технические решения

### Почему TaskIQ, а не Celery?

TaskIQ полностью async-нативный — воркер работает в том же event loop, что и FastAPI:
- Единый async SQLAlchemy-движок для API и воркера (Celery потребовал бы отдельный синхронный движок только для воркера)
- Нативный `async/await` в теле задач без `asyncio.run()`-обёрток
- Воркер запускается отдельным процессом (как и положено воркеру), но переиспользует тот же подход к подключению к БД — инициализация движка происходит через хук `WORKER_STARTUP`, аналогично `lifespan` в FastAPI

### Архитектура — слоистая

```
HTTP-запрос
    ↓
API layer (FastAPI роутеры)      — валидация, маппинг ошибок, rate limiting
    ↓
Service layer (бизнес-логика)    — оркестрация, постановка задач в очередь
    ↓
Repository layer (доступ к БД)   — все запросы через SQLAlchemy
    ↓
PostgreSQL
```

Роутеры не знают про ORM, сервисы не знают про HTTP-статусы. Каждый слой отвечает строго за своё.

### Repository pattern

Все запросы к БД инкапсулированы в `BookingRepository`. Сервисный слой работает с абстракцией, а не с конкретными SQL-запросами — это позволяет подменить репозиторий на мок в тестах без поднятия реальной базы.

### Идемпотентность задачи

Гарантия на двух уровнях:

1. **TaskIQ-уровень**: задача привязана к `booking_id`, который сохраняется как `task_id` в записи брони
2. **Уровень приложения**: перед любым изменением статуса задача проверяет текущий статус брони. Если статус уже не `pending` — задача возвращает `{"status": "skipped"}` без побочных эффектов

Это значит, что повторный запуск задачи с тем же `booking_id` (например, при retry после сбоя) никогда не создаёт дублирующую обработку и не "перезатирает" уже финализированный статус.

### Симуляция отказа внешнего сервиса

15% задач завершаются с `ExternalServiceError` (настраивается через `TASK_FAILURE_PROBABILITY` в `.env`). При отказе статус брони меняется на `failed` с записью причины в `failure_reason`. TaskIQ автоматически повторяет упавшую задачу (`retry_on_error=True`, `max_retries=3`).

### Rate Limiting

`POST /bookings` ограничен `RATE_LIMIT_PER_MINUTE` запросами с одного IP в минуту (по умолчанию 10). При превышении — `429 Too Many Requests`.

### Structured Logging

В `development` — цветной консольный вывод для удобства чтения. В `production` — JSON-строки, готовые к приёму любым агрегатором логов (ELK, Loki, Datadog). Каждая запись содержит контекстные поля: `booking_id`, `task_id`, `status`.

### Почему UUID, а не auto-increment int для ID

UUID непредсказуем (защита от перебора чужих записей через инкремент в URL), не требует централизованной координации при генерации, и используется как естественный идентификатор задачи в очереди — что прямо упрощает механизм идемпотентности.

## Структура проекта

```
booking-service/
├── src/app/
│   ├── api/v1/bookings.py      # FastAPI роутеры
│   ├── core/                   # Конфиг, логирование, исключения, lifespan
│   ├── db/                     # SQLAlchemy модели, фабрика сессий

│   ├── schemas/                 # Pydantic схемы запросов/ответов
│   └── workers/                 # TaskIQ брокер + задачи
├── migrations/                  # Alembic миграции
├── tests/
│   ├── api/                     # Тесты эндпоинтов (17 тестов)
│   └── workers/                 # Тесты воркера (6 тестов)
├── .github/workflows/ci.yml     # GitHub Actions: lint + types + tests
├── docker-compose.yml           # Продовый стек
├── docker-compose.dev.yml       # Стек для разработки (hot reload)
├── docker-compose.test.yml      # Изолированный стек для тестов
├── Dockerfile                   # Multi-stage сборка (builder / production / test)
└── Makefile                     # Команды разработчика (make help)
```
